1. Overview & source map

All item parsing lives in src/d2/items.ts. Every other module either calls into it or handles the surrounding context (save-file header, stash wrapper, merc/corpse containers).

Entry pointCalled byDescription
readItems(reader, version, constants, config, char?)char, merc, corpse, stashReads "JM" header + count, then loops over readItem
readItem(reader, version, constants, config, parent?)readItems, golem, socketed childrenReads one complete item (simple bits + optional extended data)
_readSimpleBits(item, reader, version, constants, config)readItemReads the fixed-ish header block: flags, version, location, type code
_readMagicProperties(reader, constants)readItemReads one property list (terminated by sentinel 0x1ff)

The constants object (loaded from src/data/versions/99_constant_data.ts) provides all lookup tables: magical_properties, armor_items, weapon_items, other_items, stackables, runewords, set/unique item names, etc. The sB, sP, and sA fields of each magical property entry drive the bit widths at parse time.

Version gate: This document covers only version 0x63 (99) — D2R 2.4+. Where the code branches on version >= 0x61 or version > 0x61, the v99 branch is highlighted.

2. The BitReader: LSB-first bit stream

The BitReader (src/binary/bitreader.ts) converts the raw byte array into an array of individual bits, LSB-first inside each byte. For example, byte 0x4A = 01001010 becomes the bit sequence [0,1,0,1,0,0,1,0] (bit 0 = LSB, bit 7 = MSB).

// Construction
const reader = new BitReader(buffer);   // buffer: ArrayBuffer | Uint8Array

// Reading primitives  (always LSB-first)
reader.ReadBit()          → 0 or 1
reader.ReadUInt8(n)       → reads n bits into a uint8 (n ≤ 8)
reader.ReadUInt16(n)      → reads n bits into a uint16-LE (n ≤ 16)
reader.ReadUInt32(n)      → reads n bits into a uint32-LE (n ≤ 32)
reader.ReadString(bytes)  → reads bytes*8 bits as ASCII
reader.Align()            → rounds offset up to next byte boundary

The offset field is always measured in bits, not bytes.

Example: reading a 3-bit value

If the current offset is at bit 32 and the next three bits are [1,0,1], then ReadUInt8(3) returns 0b101 = 5. The offset advances to 35.

3. Item list wrapper (readItems)

Every item list — character inventory, merc, corpse, stash page — begins with the same 4-byte wrapper:

// readItems()
"JM"              — 2 bytes (ASCII "JM" = 0x4A 0x4D), read as string
count             — uint16-LE, number of items that follow

After that, readItem is called count times sequentially. Items are not length-prefixed; each item's parser must advance the bit stream to the exact end of that item before returning.

No per-item "JM" header in D2R (v≥0x61): Classic save files (v≤0x60) have a 2-byte "JM" header before every individual item. D2R removed it. The list-level "JM" is still present.

4. Phase 1 — simple bits (_readSimpleBits)

The first thing readItem does (after the v≤0x60 per-item header check) is call _readSimpleBits. This function reads the fields that are present on every item regardless of quality or complexity.

4.2 Version & location fields

Immediately after the flags:

location_id30=stored, 1=equipped, 2=belt, 4=cursor, 5=socketed item

These fields always appear at bits 35–52 of the item data (counting from the first flag bit), regardless of item content. This is the key fact exploited by the move-without-parsing approach described in the companion document.

4.3 Ear path (is_ear === 1)

If is_ear is set, the item is a player ear (trophy). There is no item type code. Instead:

class   — uint8(3)   character class (0=Amazon, …)
level   — uint8(7)   character level
name    — up to 15 × uint8(7)   null-terminated, 7-bit chars

After the ear name, _readSimpleBits returns. Because simple_item is also always 1 for ears, the extended section is skipped.

4.4 Huffman item type code (v99 branch)

Classic D2 stored the 4-character item type code as 4 raw ASCII bytes (32 bits). D2R replaced this with a Huffman-coded representation, saving typically 8–24 bits per item.

The code reads exactly 4 characters. For each character it walks the Huffman tree one bit at a time until it lands on a leaf (a non-array node):

for (let i = 0; i < 4; i++) {
  let node = HUFFMAN_TREE;          // root of the nested-array tree
  do {
    node = node[reader.ReadBit()];  // 0 = left child, 1 = right child
  } while (Array.isArray(node));
  item.type += node;                // leaf is the character
}
item.type = item.type.trim();

Character lengths range from 2 bits (space, which pads short type codes) to 9 bits ('j').

4.5 Socket count (simple field)

After decoding the type code, the item type is looked up in the constants tables to determine its category. Then a small integer is read whose bit width depends on two conditions:

  • If the item is a Quest item: first read a quest difficulty value using constants.magical_properties[356].sB bits (with offset sA), then read socket count as 1 bit.
  • If simple_item === 1: socket count is 1 bit.
  • Otherwise (normal extended item): socket count is 3 bits.

This field (nr_of_items_in_sockets) is the count of socketed gem/rune/jewel child items appended after this item's byte-aligned end.

5. Phase 2 — extended data (non-simple items)

If simple_item === 0 and is_ear === 0, the parser continues reading the extended data section. This section has many conditional sub-fields and must be read in strict order.

5.2 Quality-specific blocks in detail

Low quality (1)
low_quality_id  — uint8(3)   // 0=crude, 1=cracked, 2=damaged, 3=low quality
Superior quality (3)
file_index  — uint8(3)   // superior type index (enhances base stats)
Magic quality (4)
magic_prefix  — uint16(11)   // index into magic_prefixes table; 0 = no prefix
magic_suffix  — uint16(11)   // index into magic_suffixes table; 0 = no suffix
Set quality (5)
set_id  — uint16(12)   // index into set_items table
Unique quality (7)
unique_id  — uint16(12)   // index into unq_items table
Rare (6) / Crafted (8) quality
rare_name_id   — uint8(8)    // first rare name word index
rare_name_id2  — uint8(8)    // second rare name word index

// Then 6 optional affix slots (3 prefixes + 3 suffixes, interleaved):
for i in 0..5:
  has_affix  — ReadBit()
  if has_affix:
    magical_name_ids[i]  — uint16(11)   // affix ID
  else:
    magical_name_ids[i]  = null

A rare item can have up to 3 prefixes and 3 suffixes. The 6 slots are: prefix0, suffix0, prefix1, suffix1, prefix2, suffix2.

5.3 Runeword block (given_runeword === 1)

If the runeword flag was set in the flags word, read immediately after the quality block:

runeword_id  — uint16(12)   // index into runewords table
               // Special case: raw ID 2718 is remapped to 48 (Delirium)
padding      — uint8(4)     // always 5 in practice; read and discard

5.4 Personalized name (personalized === 1)

If the personalized flag was set, read a null-terminated character name (max 15 chars + null = 16 total):

// v99 (version > 0x61): 8-bit chars
// v96/v97 (version <= 0x61): 7-bit chars
for i in 0..15:
  ch  — uint8(8)      // 8-bit ASCII in v99
  if ch == 0x00: break
personalized_name = decoded string
D2R (v≥0x62) uses full 8-bit characters for personalized names. Earlier D2R patches (v97/v98, 0x61) and classic D2 (v96, 0x60) used 7-bit characters.

5.5 Tome flag & realm timestamp

// Tome of Town Portal or Tome of Identify
if item.type == "tbk" or "ibk":
  tome_flag  — uint8(5)   // 0 = TP tome, 1 = ID tome (always 0 or 1)

// Realm data: always present for non-simple extended items
timestamp  — uint8(1)   // 0 = not a realm item, 1 = realm item

5.6 Defense rating & durability

The presence of these fields depends on the item type category determined from the constants tables:

// Only for armor (type_id == 1) and shields (type_id == 2, treated as armor):
defense_rating  — uint16(sB[31]) - sA[31]
  // stat ID 31 controls defense; sB = bit width, sA = subtracted offset

// For armor AND weapons:
max_durability  — uint16(sB[73]) - sA[73]    // stat ID 73
if max_durability > 0:
  current_durability  — uint16(sB[72]) - sA[72]  // stat ID 72

In the v99 constants, magical_properties[72].sB = 9, magical_properties[73].sB = 9, and magical_properties[31].sB = 11. The sA offsets are subtracted on read (and added back on write) to allow signed ranges to be stored as unsigned.

5.7 Quantity & socket count

// Only if item.type is in the stackables table (arrows, bolts, potions, scrolls, keys…)
quantity  — uint16(9)   // stack size, 0–511

// Only if socketed flag was set in the flags word:
total_nr_of_sockets  — uint8(4)   // how many sockets the item has (0–6)

5.8 Set plist flag

// Only for Set quality items:
plist_flag  — uint8(5)
// Each set bit means one additional bonus property list follows after
// the main magic_attributes list. The bit count determines set_list_count.

The plist flag is a bitmask: bit 0 set = 1 bonus list, bits 0+1 set = 2 bonus lists, etc. These bonus lists encode the partial-set bonuses that activate when wearing multiple pieces of the same set.

6. Phase 3 — magic property lists (_readMagicProperties)

After the Phase 2 fields, one or more property lists are read. Each list is a sequence of property entries terminated by the sentinel value 0x1ff.

6.1 Basic property

id  — uint16(9)    // 9-bit stat ID from ItemStatCost.txt
if id == 0x1ff: end of list

// Stat value
v  — uint16(sB[id])    // sB = save bits for this stat
if sA[id]:
  v -= sA[id]          // adjust for signed/offset encoding

6.2 Multi-stat group (np field)

Some stats are grouped: e.g., "+5 to All Resistances" is stored as a single ID but covers 4 stats. When constants.magical_properties[id].np > 1, the parser reads np consecutive entries from id through id + np - 1. All sub-entries share the same id in the output but have their own sB/sA/sP fields.

np = magical_properties[id].np || 1    // group size, usually 1
for j in 0..np-1:
  prop = magical_properties[id + j]
  // read sP, sB for each sub-stat

6.3 Parameter field (sP)

Some stats carry a parameter (e.g., which skill to add levels to, or which element for damage). When prop.sP > 0, the parameter is read before the value:

if prop.sP:
  param  — uint16(sP)    // raw parameter bits
  // dF == 14: +skill to skill tab
  if prop.dF == 14:
    skill_tab  = param & 0x7
    param      = (param >> 3) & 0x1fff
    values.push(skill_tab)
  // encoding 2 (chance to cast) or 3 (charges): split param
  if prop.e == 2 or 3:
    values.push(param & 0x3f)    // skill level
    param = (param >> 6) & 0x3ff // skill ID
  values.push(param)

6.4 Encoding types (e field)

The e field on a magical property entry selects a special decoding for the value bits:

eNameDecode logic
0Normalvalues.push(v)
1UnimplementedTreated as normal (no special decode)
2Chance to castparam → skill level (sP & 0x3f) + skill ID (sP >> 6); value → raw damage
3Chargesparam → skill level + skill ID; value bits split: low 8 = current charges, high 8 = max charges

Example: Charges (e=3)

// "Level 3 Teleport (27/27 charges)"
// sP = some bits → param = skill_level=3, skill_id=37 (Teleport)
// sB bits → v = (27 << 8) | 27
values = [3, 37, 27, 27]   // [param_skill_level, param_skill_id, current, max]

6.5 Sentinel value

Each property list is terminated by reading a 9-bit value equal to 0x1ff (all 9 bits set = 511 decimal). The library reads the stat ID first, checks against 0x1ff, and exits the loop if matched.

6.6 List ordering for non-normal qualities

After reading the main magic_attributes list, additional lists may follow:

ConditionAdditional lists
Quality = Set and plist_flag has N bits setN extra lists → set_attributes[0..N-1]
given_runeword === 11 extra list → runeword_attributes

The runeword property list always comes last, after any set attribute lists.

7. Byte alignment

After all extended data and property lists have been read (or after simple bits if simple_item === 1), the bit reader is aligned to the next byte boundary:

reader.Align();   // offset = (offset + 7) & ~7

This is done once per item, not once per list. The padding bits are discarded.

Byte alignment happens before reading socketed children. The socketed children are full items in their own right — each starts at a fresh byte boundary and ends with its own alignment step.

8. Socketed children

After the parent item is byte-aligned, any socketed items (gems, runes, jewels) are read:

if nr_of_items_in_sockets > 0 and simple_item === 0:
  socketed_items = []
  for i in 0..nr_of_items_in_sockets-1:
    socketed_items.push(readItem(reader, version, constants, config, parentItem))

Socketed items are always simple (simple_item === 1). They have no extended data, no property lists of their own in the binary stream — their properties are calculated at runtime by the attribute enhancer based on the parent item's type and the gem/rune type. This is why nr_of_items_in_sockets (the number of filled sockets) is different from total_nr_of_sockets (the total number of socket holes on the item).

9. Complete parsing flow

The following pseudocode mirrors the actual library code in items.ts, with all branches made explicit:

function readItem(reader, version, constants):
  item = {}

  // ── Phase 1: simple bits ────────────────────────────
  item.b0_3          = reader.ReadBitArray(4)
  item.identified    = reader.ReadBit()
  item.b5_10         = reader.ReadBitArray(6)
  item.socketed      = reader.ReadBit()
  item.b12           = reader.ReadBitArray(1)
  item.new           = reader.ReadBit()
  item.b14_15        = reader.ReadBitArray(2)
  item.is_ear        = reader.ReadBit()
  item.starter_item  = reader.ReadBit()
  item.b18_20        = reader.ReadBitArray(3)
  item.simple_item   = reader.ReadBit()
  item.ethereal      = reader.ReadBit()
  item.b23           = reader.ReadBitArray(1)   // IFLAG_JUSTSAVED
  item.personalized  = reader.ReadBit()
  item.b25           = reader.ReadBitArray(1)   // IFLAG_LOWQUALITY
  item.given_runeword= reader.ReadBit()
  item.b27_31        = reader.ReadBitArray(5)

  item.version       = reader.ReadUInt16(3).toString(2)  // v99: 3 bits

  item.location_id   = reader.ReadUInt8(3)   // ← always at bit 35
  item.equipped_id   = reader.ReadUInt8(4)
  item.position_x    = reader.ReadUInt8(4)
  item.position_y    = reader.ReadUInt8(4)
  item.alt_position_id = reader.ReadUInt8(3)

  if item.is_ear:
    item.ear_attributes.class = reader.ReadUInt8(3)
    item.ear_attributes.level = reader.ReadUInt8(7)
    item.ear_attributes.name  = readNullTerminatedString(reader, 15, 7)
    // simple_item is always 1 for ears → skip Phase 2
  else:
    // Huffman type decode (v99)
    item.type = ""
    for 0..3:
      node = HUFFMAN_TREE
      do: node = node[reader.ReadBit()] while isArray(node)
      item.type += node
    item.type = item.type.trim()

    item.categories = lookupItemCategories(item.type, constants)
    item.type_id    = categorize(item.categories)  // Armor=1, Weapon=3, Other=4

    if item.categories includes "Quest":
      item.quest_difficulty = reader.ReadUInt16(sB[356]) - sA[356]
      item.nr_of_items_in_sockets = reader.ReadUInt8(1)
    else if item.simple_item:
      item.nr_of_items_in_sockets = reader.ReadUInt8(1)
    else:
      item.nr_of_items_in_sockets = reader.ReadUInt8(3)

  // ── Phase 2: extended data (skip if simple_item) ────
  if not item.simple_item:
    item.id                = reader.ReadUInt32(32)
    item.level             = reader.ReadUInt8(7)
    item.quality           = reader.ReadUInt8(4)
    item.multiple_pictures = reader.ReadBit()
    if item.multiple_pictures:
      item.picture_id      = reader.ReadUInt8(3)
    item.class_specific    = reader.ReadBit()
    if item.class_specific:
      item.auto_affix_id   = reader.ReadUInt16(11)

    // Quality-specific block
    switch item.quality:
      case Low:      item.low_quality_id = reader.ReadUInt8(3)
      case Normal:   (nothing)
      case Superior: item.file_index     = reader.ReadUInt8(3)
      case Magic:    item.magic_prefix   = reader.ReadUInt16(11)
                     item.magic_suffix   = reader.ReadUInt16(11)
      case Set:      item.set_id         = reader.ReadUInt16(12)
      case Unique:   item.unique_id      = reader.ReadUInt16(12)
      case Rare,
      case Crafted:  item.rare_name_id   = reader.ReadUInt8(8)
                     item.rare_name_id2  = reader.ReadUInt8(8)
                     for i in 0..5:
                       if reader.ReadBit():
                         item.magical_name_ids[i] = reader.ReadUInt16(11)
                       else:
                         item.magical_name_ids[i] = null

    // Runeword block
    if item.given_runeword:
      item.runeword_id = reader.ReadUInt16(12)
      if item.runeword_id == 2718: item.runeword_id = 48   // Delirium fix
      reader.ReadUInt8(4)   // 4 padding bits (always 5)

    // Personalized name
    if item.personalized:
      arr = new Uint8Array(16)
      for i in 0..15:
        arr[i] = reader.ReadUInt8(8)   // 8-bit chars in v99
        if arr[i] == 0: break
      item.personalized_name = decode(arr)

    // Tome
    if item.type == "tbk" or "ibk":
      reader.ReadUInt8(5)   // tome sub-type flag

    // Realm timestamp
    item.timestamp = reader.ReadUInt8(1)

    // Defense rating (armor only)
    if item.type_id == Armor:
      item.defense_rating = reader.ReadUInt16(sB[31]) - sA[31]

    // Durability (armor + weapon)
    if item.type_id == Armor or Weapon:
      item.max_durability = reader.ReadUInt16(sB[73]) - sA[73]
      if item.max_durability > 0:
        item.current_durability = reader.ReadUInt16(sB[72]) - sA[72]

    // Quantity (stackables)
    if constants.stackables[item.type]:
      item.quantity = reader.ReadUInt16(9)

    // Socket count
    if item.socketed == 1:
      item.total_nr_of_sockets = reader.ReadUInt8(4)

    // Set plist flag
    plist_flag = 0
    if item.quality == Set:
      plist_flag = reader.ReadUInt8(5)

    // ── Phase 3: property lists ──────────────────────
    item.magic_attributes = readMagicProperties(reader, constants)

    // Set bonus lists
    while plist_flag > 0:
      if plist_flag & 1:
        item.set_attributes.push(readMagicProperties(reader, constants))
      plist_flag >>>= 1

    // Runeword attributes
    if item.given_runeword == 1:
      item.runeword_attributes = readMagicProperties(reader, constants)

  // ── Byte align ──────────────────────────────────────
  reader.Align()

  // ── Socketed children ───────────────────────────────
  if item.nr_of_items_in_sockets > 0 and not item.simple_item:
    for i in 0..nr_of_items_in_sockets-1:
      item.socketed_items.push(readItem(reader, version, constants))

  return item

10. Write path notes

The write path (writeItem, _writeSimpleBits, _writeMagicProperties) mirrors the read path exactly in field order and bit widths. A few details worth knowing:

  • Version quirk: writeStashSection writes version 0x62 regardless of the input version, so re-saved stash files will downgrade from v99 to v98. Item parsing for v99 and v98 is identical (both use the >= 0x61 branch).
  • Checksum: The save file checksum (a simple 32-bit sum over all bytes with a circular left-shift) must be recomputed after any modification. The library handles this in src/d2/header.ts.
  • Runeword padding: When writing, the 4 padding bits after the runeword ID are always written as 5 (the constant 0x5), matching what the game expects.
  • Rare name lookup: The write path uses a fuzzy prefix-match (startsWith) to find rare name IDs from name strings, accommodating known inconsistencies in the source data.
  • IFLAG_JUSTSAVED (bit 23): Always written as 1. The library note says "always 1? IFLAG_JUSTSAVED".
