D2R Item Binary Format

Save file version: 99 (0x63) — Diablo II: Resurrected 2.4+
Source: Analysis of @dschu012/d2s TypeScript library & game data files

Items in D2R are stored as a little-endian packed bitstream. Fields are not byte-aligned — they sit back-to-back at the bit level. Only the very end of each item is rounded up to a byte boundary. Every value is stored in LSB-first (least significant bit first) order within each byte.

1. High-Level Structure

Every item consists of up to three parts:

SIMPLE BITS
Every item. Flags + location + type.
EXTENDED DATA
Only if simple_item == 0
SOCKETED CHILDREN
Only if nr_of_items_in_sockets > 0

The bitstream for simple bits + extended data is terminated with a byte-alignment pad (0–7 zero bits to reach the next byte boundary). Socketed child items follow after that, each also a complete item bitstream with its own alignment.

D2R vs Classic D2 (v96): In D2R (version ≥ 0x61), individual items do not have a "JM" (0x4A 0x4D) header prefix. Only the item list starts with "JM". In classic D2 every individual item started with "JM".

2. Item List Container

Items are always grouped inside an item list:

"JM"          2 bytes  ASCII marker (0x4A 0x4D)
item_count    2 bytes  uint16 little-endian, number of items following
item[0]       variable byte-aligned item bitstream
item[1]       ...
item[N-1]     ...

Item lists appear in multiple places within a save file:

Corpse items"JM" + uint16 corpse_count; then per corpse: 12 bytes + "JM" item list12 bytes = unk(4) + x_pos(4) + y_pos(4)
Mercenary itemsSection header "jf" (2 bytes), then "JM" item listOnly present when merc exists

3.2 Item Version (3 bits, bits 32–34)

In D2R (save file version ≥ 0x61), the item version stored inside the item itself is 3 bits, read as a binary string. The most common observed value is 101 (binary for 5).

Classic D2 difference: In save version 96 (0x60 = D2 LoD 1.10–1.14d), the item version was a 10-bit decimal value: 0 = pre-1.08, 1 = 1.08/1.09 normal, 2 = 1.10 normal, 100 = expansion 1.08/1.09, 101 = expansion 1.10.

Significance for item moving: These 18 bits at a fixed offset (bits 35–52) are the only fields that need to change when repositioning an item. Flags (bits 0–31), version (32–34), and everything after bit 52 remain untouched.

3.4 Item Type Code (Huffman-encoded, bits 53+)

In D2R, the four-character item type code (e.g., hax for Hand Axe, rin for Ring, r01 for El Rune) is stored using Huffman coding rather than 4 raw ASCII bytes. Each character is decoded by traversing a binary tree, reading one bit at a time (0 = go left, 1 = go right) until reaching a leaf node that contains the character. This is done four times to recover all four characters.

Because different characters have different path lengths, the total bits for the type code varies: from 8 bits (e.g., two 4-bit characters) up to 36 bits (four 9-bit characters like j). Common codes like rin (ring) or hax (hand axe) use around 17–19 bits.

Classic D2 difference: In version 96 (0x60), the type was always 4 raw ASCII bytes = 32 bits. In D2R it is variable length (8–36 bits). This means you cannot skip the type field without either decoding it or having the Huffman tree available.

Huffman Tree Structure

The tree below shows the full decoding tree. Characters with shorter paths are more frequent in type codes. Reading 0 goes left, 1 goes right.

Example: Encoding "hax " (Hand Axe)

Char  Path          Bits  Value (LSB first)
'h'   left×4,right  5b    00011
'a'   right×4,left  5b    11110  (well: 1→1→1→1→0 = 11110)
'x'   left×2,right,left,right,left  5b  00111
' '   right,left    2b    10

Total: 5+5+5+2 = 17 bits  (classic D2 always used 32 bits for this)

Example: Encoding "r01 " (El Rune)

Char  Bits  Notes
'r'   5b    common character
'0'   8b    rare character — long path
'1'   7b    moderate
' '   2b    space always 2 bits

Total: 5+8+7+2 = 22 bits

3.5 Nr of Socketed Children / Ear Data / Quest Items

After the type code, parsing branches based on item type:

ConditionWhat is read
is_ear == 1 Instead of a type code + socket count, reads:
class (3 bits) + level (7 bits) + null-terminated name in 7-bit characters (up to 15 chars). No socket count, no extended data.
Item category includes "Quest" quest_difficulty (2 bits, using magical_properties[356]), then nr_of_items_in_sockets as 1 bit
simple_item == 1 nr_of_items_in_sockets as 1 bit
simple_item == 0 (complex/non-simple) nr_of_items_in_sockets as 3 bits
nr_of_items_in_sockets is the count of gems/runes/jewels currently inserted into this item. It is not the total number of socket holes (that is stored later in total_nr_of_sockets).

4. Extended Data — Non-Simple Items Only

If simple_item == 0, the following extended data immediately follows the simple bits (no gap, still bit-packed). Simple items (potions, scrolls, keys, gold, unsocketed gems) end at the byte-aligned boundary after their simple bits.

Index into the MagicPrefix / MagicSuffix tables. A value of 0 means no prefix or suffix on that slot.

Example: "Merciless War Axe of Quickness"

magic_prefix = 562  (11 bits) -- "Merciless" (+300-340% Enhanced Damage)
magic_suffix = 2    (11 bits) -- "of Quickness" (+20% Increased Attack Speed)

12 bits

Index into the SetItems data table, identifying which specific set item this is (e.g., Tal Rasha's Lidless Eye).

4.3 Runeword ID (16 bits, conditional)

Present only when given_runeword == 1 in the flags:

runeword_id
12 bits
padding
4 bits (always 5)

The runeword_id indexes into the Runes table. The 4-bit padding field appears to always be written as 5. (Exception: Delirium runeword has a known quirk where ID 2718 must be remapped to 48.)

4.4 Personalized Name (conditional)

Present only when personalized == 1 in the flags:

A null-terminated string stored in 8-bit characters (D2R version > 0x61). Maximum 16 characters including the null terminator. 

for i = 0 to 15:
    char_byte = read 8 bits
    if char_byte == 0x00:
        break
    name += char_byte
Classic D2 difference: Version ≤ 0x60 (classic D2) used 7-bit characters for the personalized name. D2R (version > 0x61) uses 8-bit characters.

4.5 Tome Data & Timestamp

Tome data (5 bits, conditional): Only present when the item type is tbk (Tome of Town Portal) or ibk (Tome of Identify). The 5-bit value distinguishes the two tome types at the binary level.

Timestamp / realm data (1 bit): Always present for non-simple items. This single bit is used as a realm data flag.

4.6 Base Stats (Defense, Durability, Quantity)

These fields are conditionally present based on the item type category:

Defense Rating — Armor items only

defense_rating (raw)
11 bits (sB=11)
raw_value     = read 11 bits
defense_rating = raw_value - 10        (sA = 10, stat ID 31: armorclass)

The offset of 10 allows representing a defense of 0 (raw = 10) without needing a sign bit. This field uses the bit-width from magical_properties[31].sB = 11.

Example: Archon Plate base defense

11-bit raw value = 487
defense_rating   = 487 - 10 = 477

Durability — Armor & Weapon items

max_durability
8 bits (sB=8)
current_durability
9 bits (if max_dur > 0)
max_durability  = read 8 bits   (stat ID 73: maxdurability, sB=8, sA=0)
if max_durability > 0:
    current_durability = read 9 bits  (stat ID 72: durability, sB=9, sA=0)

Items with max_durability == 0 are indestructible base types (phase blades, etc.) and do not have the current_durability field.

Quantity — Stackable items only

quantity
9 bits

Present only for items listed in the stackables table: arrows, bolts, throwing knives, throwing axes, and javelins.

4.7 Total Sockets (4 bits, conditional)

Present only when socketed == 1 in the flags:

total_nr_of_sockets
4 bits

The total number of socket holes in the item (0–6), regardless of how many are currently filled. This is distinct from nr_of_items_in_sockets in the simple bits which counts the filled sockets.

4.8 Set Bonus Flags (5 bits, conditional)

Present only when quality == 5 (Set item):

plist_flag
5 bits

A 5-bit bitmask. Each set bit indicates one additional set-bonus property list follows after the main magic properties. The number of extra lists equals the popcount (number of set bits) of this field. 

plist_flag = read 5 bits
extra_list_count = popcount(plist_flag)

// Example: plist_flag = 0b00111 = 3 extra lists
//          wearing 2-piece, 3-piece, and 4-piece bonuses are active

5. Magic Properties

Magic properties encode all item modifiers: stat bonuses, skill bonuses, on-hit effects, and more. They are stored as a variable-length list terminated by a 9-bit sentinel value 0x1FF (decimal 511). Each entry starts with the 9-bit stat ID, then reads a variable number of additional bits determined by the stat definition from ItemStatCost.txt.

5.1 Basic Property Encoding

while true:
    stat_id = read 9 bits
    if stat_id == 0x1FF:   // 511 = terminator
        break

    prop = magical_properties[stat_id]   // look up bit widths from game data

    if prop.sP is defined:               // save param bits
        param = read prop.sP bits
        // decode param (see §5.3 and §5.4)

    raw_value = read prop.sB bits        // save bits
    value = raw_value - prop.sA          // save add (signed offset)
stat_id
9 bits (0x000–0x1FE valid, 0x1FF = end)
param
sP bits (only if sP defined)
value (raw)
sB bits

Key fields in each property definition (ItemStatCost.txt):

sB — Save Bits
Number of bits used to store the stat value. Varies per stat (6–16 bits typically).
sA — Save Add
Offset subtracted from the raw value after reading. Encodes signed ranges without explicit sign bits. E.g. sA=32 means a raw value of 32 represents an actual value of 0.
sP — Save Param Bits
Extra parameter bits before the value (e.g. skill ID for "chance to cast" effects).
e — Encoding
Special encoding type: 0/absent = standard, 2 = chance-to-cast, 3 = charges.
np — Num Properties
How many consecutive stat slots this one ID covers. E.g. fire damage reads min+max under one ID.
dF — Desc Func
Display formatting function. dF=14 has special param decoding for skill-tab bonuses.

Example: "+20 to Strength" (stat ID 0)

stat_id       = 0          (9 bits)   → "strength"
prop.sB       = 8          → read 8 bits
prop.sA       = 32         → subtract 32

raw_value     = 52
actual_value  = 52 - 32 = 20

Stored in binary: 000000000 (stat_id=0) + 00110100 (raw 52)
                  = 17 bits total for this property

Example: "+50 to Life" (stat ID 7, maxhp)

stat_id       = 7          (9 bits)   → "maxhp"
prop.sB       = 9          → read 9 bits
prop.sA       = 32         → subtract 32

raw_value     = 82
actual_value  = 82 - 32 = 50

Total bits: 9 (stat_id) + 9 (value) = 18 bits

Example: "10% Enhanced Defense" (stat ID 16)

stat_id       = 16         (9 bits)   → "item_armor_percent"
prop.sB       = 9          → read 9 bits
prop.sA       = 0          → no offset

raw_value     = 10
actual_value  = 10

Total bits: 9 + 9 = 18 bits

Some stats consume multiple consecutive stat ID slots. A single stat_id read triggers reading np sub-entries (using IDs stat_id, stat_id+1, …, stat_id+np-1). Each sub-entry reads its own sB bits. No additional stat_id reads happen for the sub-entries.

  sub-entry 1 (ID 55): sB=8, sA=0 → read 8 bits = 129  (cold max)
  sub-entry 2 (ID 56): sB=8, sA=0 → read 8 bits = 100  (length in frames; 100/25 = 4 sec)

5.3 Encoding 2: Chance to Cast / Skill on Event

Stats with e=2 (e.g., stat IDs 195–201: item_skillonattack, item_skillonhit, etc.) pack skill identity and probability together: 

stat_id    = read 9 bits       (e.g. 198 = item_skillonhit)
// prop.sP = 16, prop.sB = 7, e = 2

param      = read 16 bits (sP)
skill_level = param & 0x3F          // lower 6 bits
skill_id    = (param >> 6) & 0x3FF  // next 10 bits

chance     = read 7 bits (sB)       // percentage (raw - sA)

Example: "5% Chance to Cast Level 3 Chain Lightning on Striking"

stat_id = 198 (9 bits, item_skillonhit)
param   = 0x00C3 (16 bits)
  skill_level = 0xC3 & 0x3F = 3
  skill_id    = (0xC3 >> 6)  = 3   (Chain Lightning)
chance  = 5 (7 bits)

Total bits: 9 + 16 + 7 = 32 bits

5.4 Encoding 3: Charges

Stats with e=3 (stat ID 204: item_charged_skill) pack skill identity, current charges, and max charges: 

stat_id    = read 9 bits       (204 = item_charged_skill)
// prop.sP = 16, prop.sB = 16, e = 3

param      = read 16 bits (sP)
skill_level = param & 0x3F          // lower 6 bits
skill_id    = (param >> 6) & 0x3FF  // next 10 bits

value      = read 16 bits (sB)
current_charges = value & 0xFF       // lower 8 bits
max_charges     = (value >> 8) & 0xFF // upper 8 bits

Example: "Level 12 Teleport (20/20 Charges)"

stat_id = 204 (9 bits, item_charged_skill)
param   = encodes: skill_level=12, skill_id=[Teleport ID]
value   = 0x1414 (hex): current=20, max=20

Total bits: 9 + 16 + 16 = 41 bits

5.5 Param-based Properties (sP, standard encoding)

Many properties use sP for a parameter that gives context to the value:

Skill Tab Bonus (dF = 14)

Used for "+X to Combat Skills (Paladin)":

param = read sP bits
tab   = param & 0x7          // skill tab within class (0, 1, or 2)
class = (param >> 3) & 0x1FFF // class index
value = read sB bits - sA    // bonus amount

Specific Skill Bonus

Used for "+X to Teleport (Sorceress Only)":

param    = read sP bits   // skill ID
value    = read sB bits - sA   // bonus levels

6. Property List Order

For non-simple items, magic properties appear in a specific order:

  1. Main magic properties — Always present. Terminated by 0x1FF (9 bits).
  2. Set bonus properties (0–5 additional lists) — Only for quality 5 (Set items). One list per set bit in plist_flag. Each terminated by 0x1FF.
  3. Runeword properties (1 additional list) — Only when given_runeword == 1. Terminated by 0x1FF.

Example: Set Item with 3 set-bonus lists (plist_flag = 0b00111)

[main properties...]   0x1FF   -- item's own always-active stats
[set bonus list 1...]  0x1FF   -- bonus when 2 set pieces are worn
[set bonus list 2...]  0x1FF   -- bonus when 3 set pieces are worn
[set bonus list 3...]  0x1FF   -- bonus when 4 set pieces are worn

Example: Runeword Item (Enigma: Jah + Ith + Ber)

[main properties...]        0x1FF  -- base item's inherent stats (defense etc.)
[runeword properties...]    0x1FF  -- Enigma's runeword bonuses (Teleport, str%, etc.)

7. Byte Alignment

After all property lists are written (or after simple bits for simple items), the bitstream is padded to the next byte boundary. Between 0 and 7 zero bits are inserted to ensure the next item (or the end of the stream) starts on a byte boundary. 

// After writing all item data:
padding_bits = (8 - (current_bit_offset % 8)) % 8
write padding_bits zero bits

Example

If a simple item ends at bit offset 73:
  73 % 8 = 1 → need 7 more bits to reach 80
  Padding: 7 zero bits
  Final size: 80 bits = 10 bytes
This byte alignment is critical: it means that even though the item internals are bit-packed, each item as a whole is byte-aligned. Socketed child items also start on byte boundaries.

8. Socketed Child Items

If nr_of_items_in_sockets > 0 and simple_item == 0, that many complete child items follow immediately after the parent's byte-aligned end. Each child item is fully parsed recursively using the same format. 

// After parent item byte alignment:
for i = 0 to nr_of_items_in_sockets - 1:
    parse child_item[i]     // complete item, recursively
    // child is also byte-aligned at its end

Inserted gems, runes, and jewels are typically simple items (gems, runes) or non-simple items with their own magic properties (jewels). Each has its own byte-aligned boundary.

12. Worked Example: Simple Item (Minor Healing Potion)

Let's trace the parsing of a D2R simple item — a Minor Healing Potion (hp1) stored in inventory at position (0,0).

Raw bytes (10 bytes, 80 bits)

Hex: 10 00 A0 00 05 E4 04 4F B4 00

In binary (LSB of each byte first, as the bit reader sees it):
Byte 0 (0x10): 0,0,0,0,1,0,0,0
Byte 1 (0x00): 0,0,0,0,0,0,0,0
Byte 2 (0xA0): 0,0,0,0,0,1,0,1
Byte 3 (0x00): 0,0,0,0,0,0,0,0
...
(remaining bytes encode location + huffman type + socket count + padding)

Step 1: Flags (bits 0–31)

b0–b3  (4 bits): 0000   unknown
b4     (1 bit):  0      NOT identified
b5–b10 (6 bits): 000000 unknown
b11    (1 bit):  0      NOT socketed
b12    (1 bit):  0      unknown
b13    (1 bit):  0      NOT new
b14–15 (2 bits): 00     unknown
b16    (1 bit):  0      NOT ear
b17    (1 bit):  0      NOT starter
b18–20 (3 bits): 000    unknown
b21    (1 bit):  1      ← SIMPLE ITEM
b22    (1 bit):  0      NOT ethereal
b23    (1 bit):  1      unknown (always 1)
b24    (1 bit):  0      NOT personalized
b25    (1 bit):  0      unknown
b26    (1 bit):  0      NOT runeword
b27–31 (5 bits): 00000  unknown

Step 2: Item version (bits 32–34)

3 bits: 101  → version "101" (D2R format)

Step 3: Location (bits 35–52)

location_id      (3 bits): 000 = 0  → stored in container
equipped_id      (4 bits): 0000 = 0 → not equipped
position_x       (4 bits): 0000 = 0 → column 0
position_y       (4 bits): 0000 = 0 → row 0
alt_position_id  (3 bits): 001  = 1 → inventory

Step 4: Item type code (Huffman decode, bits 53+)

Decode 4 characters using Huffman tree:
  'h': read bits until leaf → 5 bits  → 'h'
  'p': read bits until leaf → 5 bits  → 'p'
  '1': read bits until leaf → 7 bits  → '1'
  ' ': read bits until leaf → 2 bits  → ' '
  Total: 19 bits  →  type = "hp1 "  → trimmed: "hp1"

Step 5: Socket count (bit 72)

simple_item = 1, so read 1 bit:
  nr_of_items_in_sockets = 0

Step 6: Byte alignment

Bits used: 32 + 3 + 18 + 19 + 1 = 73 bits
Next byte boundary: 80 bits
Padding added: 7 zero bits
Final item size: 10 bytes

Result

type            = "hp1"  (Minor Healing Potion)
simple_item     = 1      (no extended data)
location_id     = 0      (stored)
alt_position_id = 1      (inventory)
position_x      = 0
position_y      = 0
size            = 10 bytes
