Moving Items Between Save Files

D2R Save File Format Analysis — Version 99 (0x63)

This document analyses whether it is possible to extract an item from one D2R save file and insert it into another without fully parsing all of the item's properties. The focus is on the binary format of the save file, not on how any specific library implements item handling.

See also: D2R Item Binary Format Reference

1. TL;DR — Quick Answer

Yes Change item position (x/y/container)
Location fields sit at a fixed bit offset (35–52) in every item. Patch those 18 bits and leave everything else untouched. No knowledge of the item's type, quality, or properties is needed.
Partial Find where an item ends in the bitstream

Requires the structural walk above to find the item's byte boundaries, then a raw copy of those bytes. The walk is much simpler than a full semantic parse — values are skipped, not decoded.

Not possible. The format has no length field, no item-end sentinel, and no offset table. There is no way to know where one item ends and the next begins without walking the structure.

The flags (32 bits) and item version (3 bits) always occupy the same positions. The five location fields always follow at bit 35, totalling 18 bits. These offsets are constant across every item in D2R regardless of quality, type, number of sockets, or any other property.

2.2 What This Means in Practice

To move an item from stash slot (2, 3) to inventory slot (0, 0):

  1. Find the item's start position in the bitstream (see §2.3).
  2. Patch bits 35–52 (18 bits) with the new location values.
  3. Leave all other bits untouched.

Example: Repositioning to inventory (0, 0)

// Old values (stash slot 2,3):
location_id     = 0  (000)   stored
equipped_id     = 0  (0000)  not equipped
position_x      = 2  (0010)  column 2
position_y      = 3  (0011)  row 3
alt_position_id = 5  (101)   stash

// New values (inventory slot 0,0):
location_id     = 0  (000)   stored
equipped_id     = 0  (0000)  not equipped
position_x      = 0  (0000)  column 0
position_y      = 0  (0000)  row 0
alt_position_id = 1  (001)   inventory

// Only bits 35–52 change. Everything else in the bitstream is untouched.
No knowledge of item type, quality, or magic properties is needed for this patch. The only flag worth checking first is is_ear (bit 16) — ears still have the same location fields at the same offsets, but you would normally want to confirm you have the intended item before overwriting.

2.3 Caveat: Finding the Item's Start Position

If you're working with a full save file, items are stored back-to-back in an item list with no index table: 

"JM"         2 bytes   -- item list marker
item_count   2 bytes   -- number of items (uint16 LE)
item[0]      ? bytes   -- starts at byte 4; length unknown without parsing
item[1]      ? bytes   -- starts right after item[0] ends
...
item[N-1]    ? bytes

To find item N's start offset, you must walk through items 0 through N−1 and determine where each ends. This requires the structural walk described in Part 2. Once you know an item's start position, patching its location is trivial — the offset arithmetic is fixed.

3. Part 2: Finding Item Boundaries

3.1 There Is No Length Prefix

Items have no length field, no size header, and no end-of-item sentinel. The only structural markers in the format are:

  • The byte-alignment padding at the end of each item (0–7 zero bits to reach a byte boundary)
  • The 9-bit value 0x1FF (511) which terminates each property list — not the item itself
  • The 2-byte ASCII string "JM" which is an item list header, not an item header
The byte-alignment padding is indistinguishable from regular data — you cannot scan for it. You must parse forward through the item's structure to know when the content ends and the padding begins.

To find where an item ends, you must walk its structure level by level. Here is what each level requires:

3.2 Level 1 — Simple Bits (Always Parseable, No External Data)

1
Read 32-bit flags (bits 0–31) No external data needed. Yields is_ear, simple_item, given_runeword, personalized, socketed.
2
Read 3-bit item version (bits 32–34) Always 3 bits in D2R. Always parseable.
3
Read 18-bit location data (bits 35–52) Always 18 bits. This is also where you'd write the new position if that's all you need.

After 53 bits you have everything needed to branch: is_ear, simple_item, given_runeword, personalized, socketed.

3.3 Level 2 — Type Code (Requires Huffman Tree)

If is_ear == 0, the item type code follows at bit 53. In D2R it is Huffman-encoded: each of the 4 characters is encoded with a variable number of bits (2–9 bits per character). The total length of the type code is therefore 8–36 bits and cannot be known without decoding it.

You cannot skip the type code without the Huffman tree. There is no length prefix before the code, and no fixed-bit representation to skip over. Each character must be decoded bit by bit until a leaf node is reached in the tree, four times.

If is_ear == 1, there is no type code. Instead: class (3 bits) + level (7 bits) + null-terminated name in 7-bit characters. Self-delimiting (null byte terminates the name), so no Huffman tree is needed for ears.

Once the type code is decoded, you can determine:

  • Whether the item is Armor, Weapon, or Other (needed later to know if defense/durability fields are present)
  • Whether the item is stackable (needed to know if a quantity field is present)
  • Whether it is a quest item (affects the socket count field width)

3.4 Level 3 — Socket Count and Branch

After the type code, the socket count follows:

ConditionFields to read
Quest item2 bits (quest_difficulty) + 1 bit (socket count)
simple_item == 11 bit (socket count)
simple_item == 03 bits (socket count)
Simple items end here. If simple_item == 1, byte-align to the next byte boundary and you have found the end of this item's own data. Any socketed children (gems inserted in the item) follow after the alignment, but simple items never have socketed children in practice.

3.5 Level 4 — Extended Data (Complex Items, No ItemStatCost Needed)

For complex (simple_item == 0) items, extended data follows immediately. The good news: most extended data fields before the magic properties have fixed sizes or are self-delimiting, requiring only the values already read to navigate:

3.6 Level 5 — Magic Properties (Requires ItemStatCost Bit-Width Table)

This is the section that makes a truly "blind" skip impossible without external data. Properties are stored as a variable-length list of variable-width entries: 

while true:
    stat_id = read 9 bits
    if stat_id == 0x1FF:          // sentinel: end of this property list
        break
    prop = lookup(stat_id)        // ← REQUIRES ItemStatCost.txt
    if prop.sP is defined:
        skip prop.sP bits         // param bits (e.g. skill ID)
    skip prop.sB bits             // value bits
    if prop.np > 1:               // multi-property stat
        for i = 1 to prop.np - 1:
            sub = lookup(stat_id + i)
            skip sub.sB bits
You cannot advance past a property without knowing its bit widths. Specifically, you need sB (save bits), sP (save param bits), and np (num properties) for each stat ID. These are defined in ItemStatCost.txt, embedded in this library as src/data/versions/99_constant_data.ts.

However — and this is the key distinction — you do not need to interpret the values. The task is simply: read sP + sB bits and discard them. This is fundamentally simpler than a full semantic parse:

  • You do not need sA (the signed offset for interpreting the value)
  • You do not need dF (the display function)
  • You do not need e (the encoding type, for chance-to-cast or charges)
  • You do not need any string data, affix names, or localization tables

The number of property lists to walk past depends on flags already known:

3.7 Level 6 — Byte Alignment and Socketed Children

After all property lists, the bitstream is padded to the next byte boundary (0–7 zero bits). After that alignment, any socketed child items follow sequentially — each is a complete item parsed recursively with the same rules. 

// After the last property list's 0x1FF terminator:
align to next byte boundary

// Socketed children (if any):
for i = 0 to nr_of_items_in_sockets - 1:
    walk child item recursively   // each child also byte-aligned at its end
The socketed child items (gems, runes, jewels) are themselves complete items. Gems and runes are simple items and end quickly after Level 3. Jewels are complex items and require the full Level 5 property walk. The recursion depth is at most 1 in practice (you cannot socket an item inside a socketed item).

4. Part 3: What a Minimal Move Parser Would Need

A minimal implementation capable of extracting and reinserting items between save files, with no semantic interpretation of item data:

Required Data Inputs

  1. The Huffman tree — constant, does not change between game versions. This is the same tree used by the game engine and is small enough to hardcode.
    needed: decode type code + measure its bit length
  2. Stat bit-width table — specifically only three fields per stat from ItemStatCost.txt: sB, sP, np. This table has ~500 entries and is compact. No value offsets, no display strings, no formulas.
    needed: skip past magic properties
  3. Stackable item type set — a small set of type codes (arrows, bolts, javelins, throwing weapons) indicating which items have a 9-bit quantity field.
    needed: skip quantity field conditionally
  4. Armor / weapon type classification — a map from type code to category (Armor, Weapon, Other). Determines whether defense and durability fields are present.
    needed: skip defense/durability conditionally

What the Minimal Parser Reads (but Does Not Interpret)

  • Flags: extract 5 flag bits (is_ear, simple_item, given_runeword, personalized, socketed) — no interpretation needed beyond their boolean values
  • Item version: discard; always 3 bits
  • Location fields: read and optionally overwrite; 18 bits at fixed offset
  • Type code: decode 4 characters using Huffman tree; use only to classify (armor/weapon/stackable/quest) then discard the code
  • Extended header: read quality and conditional flags to branch correctly; values are not needed for any formula
  • Quality-specific bits: advance by the correct number of bits based on quality value; raw bits discarded
  • Property list entries: read 9-bit stat ID, look up sB + sP + np, skip that many bits; the actual property values are never used

What the Minimal Parser Does NOT Need

  • sA — save add / signed offset dF — display function e — encoding type (chance-to-cast, charges)
  • affix names set / unique / runeword names localization strings
  • damage formulas stat interaction logic game balance data

5. Part 4: Conclusion

Can you move items without parsing the full item?

For changing position only: Yes, unconditionally. The location fields sit at a fixed bit offset (bits 35–52) in every item. Once you know where the item starts in the bitstream, you overwrite 18 bits and you're done. No understanding of the item's type or properties is needed.

For finding an item's bit boundaries (needed to extract a complete item and insert it into another save): A structural walk is required. You must advance through the bitstream field by field. This walk requires the Huffman tree and the ItemStatCost bit-width table. However, you only need to read the values — never interpret them. This is substantially less complex than a full semantic parse.

For blind byte copying without any parsing: Not possible. The format contains no length fields, no item-end markers, and no offset tables. Items are stored sequentially with no mechanism for random access.

Key Insight

The D2R item format is a tightly packed bitstream optimised for storage space, not for random access. Variable-length fields (Huffman-coded type, variable-width stat properties) eliminate redundant bits at the cost of requiring sequential navigation. This is the same trade-off made in compression formats like DEFLATE.

Any implementation that wants to extract or reorder items must do at least a structural walk. But the gap between a "structural walk" and a "full semantic parse" is significant: the walk only needs a compact table of bit widths, while a full parse needs name tables, display logic, damage formulas, and value-range validation.

The minimum viable "item mover" can be summarised as three components:

Minimal implementation sketch

function findItemEnd(bits, startOffset, huffmanTree, statWidths, itemTypes):
    offset = startOffset + 32 + 3  // skip flags + version
    flags = readFlags(bits, startOffset)

    // Location (could patch here):
    offset += 18

    if flags.is_ear:
        offset += 3 + 7   // class + level
        while readBits(bits, offset, 7) != 0: offset += 7
        offset += 7       // null terminator
        return byteAlign(offset)

    // Type code:
    typeName = ""
    for i = 0 to 3:
        char, bitsConsumed = huffmanDecode(bits, offset, huffmanTree)
        typeName += char
        offset += bitsConsumed
    category = itemTypes[typeName]   // armor / weapon / stackable / quest

    // Socket count:
    offset += category.isQuest ? 3 : flags.simple_item ? 1 : 3

    if flags.simple_item:
        return byteAlign(offset)

    // Extended data header (no external table needed):
    offset += 32 + 7 + 4 + 1        // id + level + quality + multi_pic
    if readBit(bits, offset-1): offset += 3   // picture_id
    offset += 1                     // class_specific
    if readBit(bits, offset-1): offset += 11  // auto_affix_id

    quality = readBits(bits, startOffset + 32 + 3 + 18 + typeBits + socketBits + 32 + 7, 4)
    offset += qualityDataSize(quality, bits, offset)

    if flags.given_runeword: offset += 16
    if flags.personalized:
        while readBits(bits, offset, 8) != 0: offset += 8
        offset += 8   // null terminator
    if typeName == "tbk" or typeName == "ibk": offset += 5
    offset += 1   // timestamp

    if category == ARMOR: offset += 11
    if category == ARMOR or WEAPON:
        maxDur = readBits(bits, offset, 8); offset += 8
        if maxDur > 0: offset += 9
    if category.isStackable: offset += 9
    if flags.socketed: offset += 4

    plistFlag = 0
    if quality == SET: plistFlag = readBits(bits, offset, 5); offset += 5

    // Skip property lists (REQUIRES statWidths table):
    offset = skipPropertyList(bits, offset, statWidths)
    for each set bit in plistFlag:
        offset = skipPropertyList(bits, offset, statWidths)
    if flags.given_runeword:
        offset = skipPropertyList(bits, offset, statWidths)

    offset = byteAlign(offset)

    // Socketed children:
    nrSockItems = /* read earlier */
    for i = 0 to nrSockItems - 1:
        offset = findItemEnd(bits, offset, huffmanTree, statWidths, itemTypes)

    return offset

function skipPropertyList(bits, offset, statWidths):
    while true:
        statId = readBits(bits, offset, 9); offset += 9
        if statId == 0x1FF: return offset
        w = statWidths[statId]
        offset += w.sP + w.sB
        for i = 1 to w.np - 1:
            offset += statWidths[statId + i].sB
