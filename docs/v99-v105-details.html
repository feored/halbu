v99 vs v105: Item Format Comparison

Scope: Binary-level differences between D2R Legacy (v99 / 0x63) and Reign of the Warlock (v105 / 0x69) save files, with emphasis on the item bitstream format.
Purpose: Groundwork for adding full item parsing support to the halbu library, which currently stores items as raw bytes.

Source quality notice: v105 differences are derived from a single incomplete open PR (dschu012/d2s PR #86) and one code review comment. No other public documentation of v105’s binary format was found as of April 2026. Each v105-specific finding is tagged with a confidence marker: Confirmed Verified in code + reviewer corroboration — Likely Consistent with code but not independently verified — Unknown Uncertain, needs further research.

1. Three Format Eras

The D2 save file format has evolved through three distinct eras, each with different item encoding:

EraSave VersionsGameKey Item Traits
Classic D2 v96 (0x60) D2 LoD 1.10–1.14d Per-item "JM" header; raw 32-bit ASCII type codes; 10-bit item version; 7-bit personalized name chars
D2R Legacy v97–v99 (0x61–0x63) D2R 2.0–2.x No per-item "JM"; Huffman-encoded type codes; 3-bit item version; 8-bit personalized name chars
RotW v105 (0x69) D2R: Reign of the Warlock Same as D2R Legacy plus 1 extra bit after durability + 1+8 bit quantity trailer. New Warlock class (ID 7)
Key takeaway: The jump from v99 to v105 is remarkably small at the item-bitstream level. Only two binary-level changes were identified in the item encoding. The overwhelming majority of the format is unchanged.

These are the header and structure differences between v99 and v105. They affect the bytes surrounding the item section, not the item bitstream itself.

2.2 Empty Item Trailer

The empty-item trailer is the byte sequence emitted when a save has no items. It contains section markers for the item list, corpse, mercenary items, and golem sections. v105 RotW saves add an extra "lf" (0x6C 0x66) section after "kf": 

// v99 Expansion (with merc)  — 17 bytes
4A 4D 00 00   // "JM" + item_count=0  (character items)
4A 4D 00 00   // "JM" + item_count=0  (corpse items)
6A 66         // "jf"                 (merc section header)
4A 4D 00 00   // "JM" + item_count=0  (merc items)
6B 66 00      // "kf" + has_golem=0

// v105 RotW (with merc)  — 23 bytes
4A 4D 00 00   // "JM" + item_count=0  (character items)
4A 4D 00 00   // "JM" + item_count=0  (corpse items)
6A 66         // "jf"                 (merc section header)
4A 4D 00 00   // "JM" + item_count=0  (merc items)
6B 66 00      // "kf" + has_golem=0
01 00         // ??? purpose unknown
6C 66         // "lf"                 (new RotW section)
00 00         // 2 bytes (empty)
The "lf" section is only present in RotW expansion saves (ExpansionType::RotW). Classic and non-RotW expansion v105 saves use the same trailer structure as v99. See src/items/mod.rs for the full set of byte sequences.

3. Item Format: What’s Unchanged

The vast majority of the item binary format is identical between v99 and v105. This section lists everything that does not change, to establish the baseline before presenting the two differences in sections 4 and 5.

3.1 Bitstream Fundamentals

  • LSB-first packed bitstream — fields sit back-to-back at the bit level, least significant bit first within each byte.
  • No per-item "JM" header — only the item list wrapper carries "JM" + uint16 count.
  • Byte alignment at item boundaries — each item (and each socketed child) ends padded with 0–7 zero bits.
  • Item list structure — "JM" + uint16 count + sequential items, used for inventory, corpse, mercenary, and stash contexts. The Iron Golem section is not a "JM" list: it uses a "kf" (2-byte) header followed by a uint8 has_golem flag; if the flag is non-zero, a single item follows directly with no count field.

  • 32-bit flags (bits 0–31) — same layout; identified (b4), socketed (b11), is_ear (b16), simple_item (b21), ethereal (b22), personalized (b24), given_runeword (b26), and all reserved bits are at the same positions.
  • 3-bit item version at bits 32–34 — unchanged.
  • 18-bit location block at bits 35–52 — fixed offset, unchanged.
  • Huffman-encoded type code — same tree, same character bit lengths (2–9 bits per char), same 4-character decode. Likely unchanged; the PR does not modify the tree.
  • Socket count branching — quest item: 2+1 bits; simple item: 1 bit; extended item: 3 bits. Unchanged.

3.3 Extended Data Fields (non-simple items, simple_item == 0)

  • id (32b), level (7b), quality (4b) — unchanged.
  • multiple_pictures (1b) + conditional picture_id (3b) — unchanged.
  • class_specific (1b) + conditional auto_affix_id (11b) — unchanged.
  • Quality-specific fields: Low (3b), Normal (0b), Superior (3b), Magic (11+11b), Set (12b), Rare/Crafted (8+8b + 6 affix slots), Unique (12b) — all unchanged.
  • Runeword block: runeword_id (12b) + padding (4b) — unchanged.
  • Personalized name: null-terminated 8-bit chars (up to 15 + null) — unchanged.
  • Tome data (5b if type is tbk/ibk) and timestamp (1b) — unchanged.
  • Defense rating (11b, armor only) — unchanged.
  • max_durability (8b) and conditional current_durability (9b) for armor and weapons — unchanged.

3.4 Remainder of Extended Data

  • Stackable quantity (9b, for arrows, bolts, throwing weapons) — unchanged.
  • total_nr_of_sockets (4b, if socketed flag) — unchanged.
  • Set plist_flag (5b, if quality == Set) — unchanged.

3.5 Magic Properties

  • 9-bit stat ID + sB/sP bits per entry — encoding unchanged.
  • 0x1FF sentinel (9 bits, value 511) terminates each property list — unchanged.
  • Property list order: main, then set bonus lists (0–5), then runeword list — unchanged.
  • Multi-property stats (np > 1) and special encodings (e=2 chance-to-cast, e=3 charges) — unchanged.

3.6 Socketed Children

  • Read recursively after the parent item’s byte alignment — unchanged.
Bottom line: If you have a working v99 item parser, supporting v105 requires only two small version-gated additions. Both are described in the next two sections.

4. Difference #1 — Extra Bit After Durability Confirmed

In v105, a single unknown bit is inserted after the durability fields and before the stackable quantity field. This only fires for extended items (simple_item == 0).

4.1 Exact Location

In v99, after reading max_durability (and optionally current_durability), the parser proceeds directly to the stackable quantity check. In v105, one additional bit is read (and discarded) between these two points: 

// v99 and v105 (shared):
if item_is_armor_or_weapon:
    max_durability = read 8 bits          // stat ID 73: sB=8
    if max_durability > 0:
        current_durability = read 9 bits  // stat ID 72: sB=9

// v105 ONLY — NEW:
if version == 105:
    unknown_bit = read 1 bit              // purpose unknown; read and preserve

// v99 and v105 (shared):
if item_type in stackables:
    quantity = read 9 bits

Purpose unknown. The bit is skipped without comment in the PR. It could be a flag for a new field, padding, or a placeholder. For round-trip fidelity in halbu, store the bit’s value in the item model even though its semantics are currently unknown.

5. Difference #2 — Quantity Trailer Before Alignment Confirmed

In v105, a 1-bit flag followed by a conditional 8-bit quantity is written after all magic property lists and before the byte-alignment step. This applies to all items — both simple (simple_item == 1) and extended.

5.1 Exact Location

// v99 and v105 (shared): property lists finished
// For extended items: after main props + set bonus lists + runeword list
// For simple items: immediately after socket count

// v105 ONLY — NEW:
if version == 105:
    has_quantity = read 1 bit
    if has_quantity:
        quantity = read 8 bits    // stack count for RotW stash stackables

// v99 and v105 (shared):
reader.Align()   // pad to next byte boundary

5.4 Reviewer Explanation

Reviewer hyperremix commented on this hunk (PR #86, Mar 11 2026):

“For Diablo II save files at version 105, item quantity is being silently discarded (SkipBits(8)) instead of parsed. This causes quantity data to be lost when reading stackable items (materials, gems, runes, etc.) from v105 saves. The following change fixes this issue and correctly parses quantities in the new stash tabs in Diablo 2: Reign of the Warlock.”

The suggested fix replaces reader.SkipBits(8) with item.quantity = reader.ReadUInt8(8).

Semantics: The 8-bit value is the stack count for stackable items stored in RotW’s new stash tabs. The 1-bit flag distinguishes items that carry a quantity from those that do not. This is a separate field from the existing 9-bit quantity inside the extended data section (used for arrows, bolts, javelins, etc.).
Applies to ALL items, including simple ones. This trailer is read outside the if (!simple_item) block in the PR. A simple potion, rune, or gem in a v105 save also has the 1-bit flag (and the 8-bit quantity if the flag is set). Factor this into the implementation: the check must not be guarded by simple_item == 0.

6. Updated Parsing Flow (v105 Annotated)

The complete v105 item parsing flow. Lines marked NEW-v105 are the only additions relative to the v99 flow in item-details.html §9. 

┌─────────────────────────────────────────────────────────────────────────────────┐
│ PARSE ITEM (v99 unchanged + v105 additions marked)                         │
│                                                                             │
│  [32 bits] Read FLAGS                                                       │
│  [3 bits]  Read ITEM VERSION                                                │
│  [3 bits]  Read location_id      ┌───────────────────────────┦             │
│  [4 bits]  Read equipped_id      │ Fixed at bits 35-52 in all versions.  │             │
│  [4 bits]  Read position_x       │ Patch these 18 bits to move an item.  │             │
│  [4 bits]  Read position_y       └───────────────────────────┮             │
│  [3 bits]  Read alt_position_id                                             │
│                                                                             │
│  IF is_ear:                                                                 │
│    [3b] ear_class  [7b] ear_level  [7b×n] ear_name (null-terminated)       │
│    → GOTO TRAILER                                                          │
│                                                                             │
│  ELSE:                                                                      │
│    [var]  Decode HUFFMAN TYPE CODE (4 chars, 8–36 bits, same tree)          │
│    Classify type: Armor / Weapon / Other / Quest / Stackable                │
│    IF Quest:      [2b] quest_difficulty + [1b] nr_of_items_in_sockets       │
│    ELIF simple:   [1b] nr_of_items_in_sockets                               │
│    ELSE:          [3b] nr_of_items_in_sockets                               │
│                                                                             │
│  IF NOT simple_item:                                                        │
│    [32b] id  [7b] level  [4b] quality                                       │
│    [1b] multi_pic; IF set: [3b] picture_id                                  │
│    [1b] class_spec; IF set: [11b] auto_affix_id                             │
│    Quality-specific data (Low=3b, Normal=0b, Superior=3b, Magic=22b,        │
│      Set=12b, Unique=12b, Rare/Crafted=16b+6×(1b+?11b))                    │
│    IF given_runeword: [12b] runeword_id + [4b] padding                     │
│    IF personalized:   [8b×n] name (null-terminated)                        │
│    IF tbk/ibk:        [5b] tome_data                                       │
│    [1b] timestamp                                                           │
│    IF Armor: [11b] defense                                                  │
│    IF Armor OR Weapon:                                                      │
│      [8b] max_durability                                                    │
│      IF max_dur > 0: [9b] current_durability                               │
│    ┌──────────────────────────────────────────────────────────────────────┐ │
│    │ NEW-v105 [1b] unknown_bit (preserve value; purpose unknown)         │ │
│    └──────────────────────────────────────────────────────────────────────┮ │
│    IF Stackable: [9b] quantity                                              │
│    IF socketed:  [4b] total_nr_of_sockets                                  │
│    IF Set:       [5b] plist_flag                                            │
│                                                                             │
│    Read MAGIC PROPERTIES list (9b stat_id + var data, until 0x1FF)         │
│    FOR each set bit in plist_flag:                                          │
│      Read SET BONUS PROPERTIES list (until 0x1FF)                           │
│    IF given_runeword:                                                       │
│      Read RUNEWORD PROPERTIES list (until 0x1FF)                            │
│                                                                             │
│ TRAILER:                                                                    │
│  ┌──────────────────────────────────────────────────────────────────────┐  │
│  │ NEW-v105 [1b] has_quantity flag (applies to ALL items)              │  │
│  │ NEW-v105 IF has_quantity: [8b] quantity (RotW stash stack count)     │  │
│  └──────────────────────────────────────────────────────────────────────┮  │
│                                                                             │
│ ALIGN:                                                                      │
│  Pad to next byte boundary (0–7 zero bits)                                 │
│                                                                             │
│  IF nr_of_items_in_sockets > 0 AND NOT simple_item:                        │
│    FOR each socketed child: PARSE ITEM (recursive)                          │
└─────────────────────────────────────────────────────────────────────────────────┘

7.2 Architecture Notes

  • The Huffman tree is a single compile-time constant. No version branching is needed for type code decoding (the tree is likely unchanged across all D2R versions).
  • The quantity trailer (Difference #2) must be read outside any if format == simple guard. It applies unconditionally to all items.
  • The unknown bit (Difference #1) is inside the extended data block and fires only when simple_item == 0 and the item has durability fields (i.e., armor or weapon). Based on the PR code it fires after all durability reads for any extended item, including items where max_durability == 0 (indestructible). Verify this against real saves.
  • For round-trip fidelity, store the unknown bit in the item model. A field such as v105_unknown_after_durability: bool keeps the bit preserved through encode/decode cycles until its semantics are understood.
  • The existing src/utils.rs read_bits/write_bits infrastructure handles all necessary bit-level I/O for these additions.
